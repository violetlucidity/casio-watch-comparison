PROBLEM #1: SVG Shapes Don't Look Like Watches
─────────────────────────────────────────────

WHAT YOU GOT:
  Simple rectangles with dimension lines
  No visual resemblance to actual watch shapes
  Missing: case bezels, crown, dial details

WHY THIS HAPPENED:
  1. OVERSIMPLIFICATION: I generated minimal SVGs to reduce code complexity
  2. SCOPE CREEP: Started with "parametric shapes" but didn't implement visual details
  3. NO DESIGN SPEC: Never defined what "watch-like" means (rectangle vs round vs rounded corners)
  4. MISSING VISUAL REFERENCE: The mockup image I generated showed detailed watches, but 
     the code only produced basic rectangles

MY MISTAKE:
  Generated a visual mockup that set expectations for detailed watch shapes, then 
  shipped code with minimal rectangles. This created a SPECIFICATION-IMPLEMENTATION GAP.

FIX FOR FUTURE:
  → BEFORE coding: Lock down visual specification
  → Get agreement on complexity level (simple vs detailed)
  → Don't generate aspirational mockups that don't match code implementation
  → If mockup shows detail X, implementation must include detail X

PROBLEM #2: SVG Dragging Doesn't Work
──────────────────────────────────────

WHAT YOU GOT:
  SVGs don't move when dragged
  No visual feedback on drag attempts

WHY THIS HAPPENED:
  1. BROWSER SANDBOX LIMITATION: Mobile browsers have different drag handling
  2. MOUSEDOWN/MOUSEMOVE: Desktop mouse events don't work well on mobile
  3. NO TOUCH EVENTS: Code only implements mouse events, not touch events
  4. EVENT DELEGATION ISSUE: Drag event listeners may not be attaching properly
  5. TRANSFORM CONFLICTS: CSS transforms on SVG might be overriding JavaScript transforms

SPECIFIC CODE ISSUE:
  svg.addEventListener('mousedown', (e) => {
      startX = e.clientX;
      startY = e.clientY;
  });
  This only works on desktop. Mobile needs touchstart/touchmove/touchend events.

MY MISTAKE:
  Assumed mousedown/mousemove would work on mobile. Didn't implement touch events.
  Tested mentally, not physically on mobile device.

FIX FOR FUTURE:
  → ALWAYS implement both mouse AND touch events for mobile
  → Test on actual mobile device, not desktop browser
  → Use pointer events (standard) instead of mouse/touch separately
  → Verify drag targets are not prevented by z-index or pointer-events issues

PROBLEM #3: Mobile UI Is Cluttered
────────────────────────────────────

WHAT YOU GOT:
  Too many sections crammed vertically
  Small fonts hard to read
  Limited space for each component
  Sidebar section visible but poorly organized

WHY THIS HAPPENED:
  1. NO MOBILE MOCKUP: Didn't design for mobile first
  2. INFLEXIBLE LAYOUT: Tried to fit desktop 3-column into mobile
  3. NO SPACE PLANNING: Didn't calculate pixel allocations per section
  4. POOR HIERARCHY: All sections equal height instead of priority-based
  5. NO USER TESTING: Never viewed on actual mobile device

MY MISTAKE:
  Created CSS "responsive" rules without actually designing for mobile.
  Just added breakpoints and media queries without rethinking the layout.
  Never viewed the result on your actual device before declaring it "fixed".

FIX FOR FUTURE:
  → START with mobile mockup (what does it look like at 480px wide?)
  → DEFINE space allocation: What gets 40% vs 20% vs 30%?
  → TEST on actual device: Don't assume breakpoints work
  → ITERATE on mobile first, then scale up to desktop

PROBLEM #4: Communication Gap Between Perplexity & You
───────────────────────────────────────────────────────

THE LOOP THAT HAPPENED:
  1. You: "Can you build this?"
  2. Me: "Yes, here's a prototype" (not tested on mobile)
  3. You: "The sidebar isn't showing"
  4. Me: "Let me add mobile CSS" (still not tested on mobile)
  5. You: "Still not working"
  6. Me: "Let me rebuild with flexbox" (still not tested on mobile)
  7. You: "SVGs don't work, UI is cluttered"

THE ROOT CAUSE:
  - I was generating code without testing on YOUR device
  - You were telling me results didn't match expectations
  - I kept making assumptions instead of asking for clarification
  - No systematic debugging approach

MY MISTAKE:
  Treated this like a code generation task, not a product design task.
  Should have built incrementally with feedback, not shipped multiple "complete" versions.

FIX FOR FUTURE:
  → Ask clarifying questions about constraints BEFORE coding
  → Build ONE feature at a time, test it, get feedback
  → Don't promise "complete rebuilds" unless absolutely necessary
  → Test assumptions out loud with the user

PROBLEM #5: Scope Unclear (What Are We Actually Building?)
───────────────────────────────────────────────────────────

WHAT I ASSUMED:
  • Simple rectangular SVG outlines
  • Mobile-responsive grid layout
  • Click-to-select, drag-to-overlay interaction
  • Auto-calculated dimension differences

WHAT YOU ACTUALLY NEEDED (inferred from feedback):
  • SVG shapes that LOOK like watches (not just rectangles)
  • Touch-friendly dragging on mobile
  • Readable mobile layout (not cramped)
  • Working interactive features

THE DISCONNECT:
  I optimized for "code simplicity" not "user experience".
  You expected "watch comparison tool" not "dimension rectangle viewer".

MY MISTAKE:
  Never asked: "What does success look like to you?"
  Never clarified: "Should SVGs be detailed or minimal?"
  Never confirmed: "Are you testing on mobile?"

FIX FOR FUTURE:
  → Get explicit success criteria before building
  → Define "done" clearly (working SVG drag? detailed shapes? specific features?)
  → Ask about constraints (mobile vs desktop? simple vs complex? browser support?)
  → Confirm test environment (what device? what browser? what screen size?)
